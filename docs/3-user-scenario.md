# User Scenario - my-todolist

## 문서 정보
- **프로젝트명**: my-todolist
- **버전**: 1.0.0 (MVP)
- **작성일**: 2026-02-11
- **관련 문서**: [PRD](./2-PRD.MD), [도메인 정의서](./1-domain-definition.md)

## 1. 개요

### 1.1 문서 목적
본 문서는 my-todolist 애플리케이션의 실제 사용자 시나리오를 정의합니다. 각 페르소나별 구체적인 사용 흐름과 예상되는 상호작용을 상세히 기술하여 개발팀이 사용자 관점에서 제품을 구현하고 테스트할 수 있도록 합니다.

### 1.2 대상 독자
- 개발팀 (프론트엔드, 백엔드)
- QA/테스터
- 제품 관리자
- UI/UX 디자이너

### 1.3 시나리오 범위
- **포함**: MVP 기능 (회원가입, 로그인, 할일 CRUD, F-001~F-105)
- **제외**: 향후 버전 기능 (검색, 필터링, 카테고리, 공유 등)
- **대상 페르소나**: 대학생, 직장인, 주부 (총 3명)

## 2. 페르소나별 사용자 시나리오

### 2.1 페르소나 1: 대학생 김민수 (23세)

#### 2.1.1 배경 상황
김민수는 컴퓨터공학과 3학년 학생으로, 이번 학기에 5개 과목을 수강하고 있으며 프로그래밍 동아리 회장도 맡고 있습니다. 과제 마감일, 시험 일정, 동아리 행사가 겹치면서 종이 메모와 카톡 메시지로 흩어진 일정 관리에 어려움을 겪고 있습니다. 친구가 추천한 간단한 할일 관리 앱을 시도해보기로 했습니다.

#### 2.1.2 목표
- 이번 주 과제 3개를 한눈에 확인
- 중간고사 일정(D-7)을 놓치지 않도록 관리
- 동아리 세미나 준비 항목 체크리스트 작성

#### 2.1.3 시나리오: "첫 사용부터 할일 완료까지"

**사전 조건**: 없음 (신규 사용자)

**예상 소요 시간**: 약 10분

**단계별 흐름**:

##### Step 1: 회원가입 (2분)
**관련 기능**: F-001

1. 브라우저에서 `http://localhost:3000` 접속
2. 랜딩 페이지에서 "회원가입" 버튼 클릭 또는 `/register` 자동 이동
3. **화면 1: 회원가입** 표시
   - Username 입력: `minsu_kim`
   - Password 입력: `MyPassword123!` (마스킹 처리됨)
   - Email 입력: `minsu.kim@university.ac.kr`
4. "회원가입" 버튼 클릭
5. **시스템 반응**:
   - API `POST /api/auth/register` 호출
   - 로딩 스피너 표시 (0.5초)
   - 성공 시 "회원가입 성공!" 메시지
   - 자동으로 로그인 화면(`/login`)으로 이동 또는 자동 로그인

**예상 결과**:
- HTTP 201 Created 응답
- users 테이블에 새 레코드 생성 (password는 bcrypt 해시)
- 로그인 화면으로 이동

**대체 흐름 (에러)**:
- 만약 `minsu_kim`이 이미 존재하면:
  - 화면 상단에 빨간색 알림: "이미 존재하는 사용자명입니다" (E-001)
  - Username 필드에 포커스 이동

##### Step 2: 로그인 (1분)
**관련 기능**: F-002

1. **화면 2: 로그인** 표시
   - Username 입력: `minsu_kim`
   - Password 입력: `MyPassword123!`
2. "로그인" 버튼 클릭
3. **시스템 반응**:
   - API `POST /api/auth/login` 호출
   - JWT 토큰 발급 (유효기간 24시간)
   - 토큰을 localStorage에 저장 (예: `token: "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."`)
   - 성공 메시지: "환영합니다, 민수님!"
   - 할일 목록 화면(`/todos`)으로 자동 이동

**예상 결과**:
- HTTP 200 OK + JWT 토큰
- `/todos` 페이지 표시

**대체 흐름 (에러)**:
- 잘못된 비밀번호 입력 시:
  - 화면 상단에 빨간색 알림: "아이디 또는 비밀번호가 일치하지 않습니다" (E-002)
  - Password 필드 초기화

##### Step 3: 첫 할일 추가 - 과제 등록 (2분)
**관련 기능**: F-101

1. **화면 3: 할일 목록** 표시
   - 상단 헤더: "나의 할일 목록" + 사용자명 "민수님" + "로그아웃" 버튼
   - 빈 상태 메시지: "할일이 없습니다. 새로운 할일을 추가해보세요!"
   - "+ 할일 추가" 버튼 (파란색, 눈에 띄는 위치)
2. "+ 할일 추가" 버튼 클릭
3. **화면 4: 할일 추가 모달** 표시
   - 제목 입력: `자료구조 과제 - 트리 구현`
   - 설명 입력: `이진 탐색 트리 insert, delete 함수 구현하기`
   - 마감일 선택: `2026-02-15` (DatePicker로 선택)
4. "저장" 버튼 클릭
5. **시스템 반응**:
   - API `POST /api/todos` 호출 (JWT 토큰 헤더 포함)
   - 로딩 스피너 표시
   - 성공 시 모달 닫힘
   - 할일 목록에 새 항목 즉시 표시

**예상 결과**:
- HTTP 201 Created + 새 todo 객체
- 목록에 할일 카드 표시:
  ```
  [ ] 자료구조 과제 - 트리 구현
      이진 탐색 트리 insert, delete 함수 구현하기
      마감일: 2026-02-15
      [수정] [삭제]
  ```

**대체 흐름 (에러)**:
- 제목 없이 "저장" 클릭 시:
  - 제목 필드 아래 빨간색 텍스트: "할일 제목은 필수 항목입니다" (E-103)
  - 제목 필드에 빨간색 테두리 표시

##### Step 4: 추가 할일 등록 (2분)
**관련 기능**: F-101

1. 다시 "+ 할일 추가" 클릭
2. **할일 2 입력**:
   - 제목: `운영체제 중간고사 준비`
   - 설명: `Chapter 1-5 복습, 연습문제 풀이`
   - 마감일: `2026-02-18`
3. "저장" 클릭
4. **할일 3 입력**:
   - 제목: `동아리 세미나 자료 준비`
   - 설명: `React Hooks 발표 자료 PPT 20장`
   - 마감일: `2026-02-20`
5. "저장" 클릭

**예상 결과**:
- 할일 목록에 총 3개 항목 표시 (최신 것이 위)

##### Step 5: 할일 완료 처리 (30초)
**관련 기능**: F-105

1. 자료구조 과제를 완료함
2. 첫 번째 할일 카드의 체크박스 클릭
3. **시스템 반응**:
   - API `PATCH /api/todos/:id/complete` 호출
   - 체크박스에 체크 표시
   - 제목에 취소선 적용: ~~자료구조 과제 - 트리 구현~~
   - 카드 배경색 약간 회색으로 변경 (시각적 피드백)

**예상 결과**:
- HTTP 200 OK
- isCompleted = true, updatedAt 갱신

##### Step 6: 할일 수정 (1분)
**관련 기능**: F-103

1. "운영체제 중간고사 준비" 항목의 "수정" 버튼 클릭
2. **화면 4: 할일 수정 모달** 표시 (기존 데이터 미리 채워짐)
3. 설명 수정: `Chapter 1-5 복습, 연습문제 풀이, 기출문제 3개년 분석`
4. 마감일 수정: `2026-02-17` (하루 앞당김)
5. "저장" 버튼 클릭
6. **시스템 반응**:
   - API `PUT /api/todos/:id` 호출
   - 모달 닫힘
   - 목록에 수정된 내용 즉시 반영

**예상 결과**:
- HTTP 200 OK + 수정된 todo 객체
- updatedAt 자동 갱신

##### Step 7: 할일 삭제 (30초)
**관련 기능**: F-104

1. "동아리 세미나 자료 준비" 항목의 "삭제" 버튼 클릭
2. **화면 5: 삭제 확인 다이얼로그** 표시
   - 메시지: "정말 삭제하시겠습니까?"
   - 할일 제목: "동아리 세미나 자료 준비"
   - [삭제] [취소] 버튼
3. "삭제" 버튼 클릭 (빨간색)
4. **시스템 반응**:
   - API `DELETE /api/todos/:id` 호출
   - 다이얼로그 닫힘
   - 해당 항목이 목록에서 사라짐 (애니메이션: 페이드아웃)
   - 성공 메시지: "할일이 삭제되었습니다"

**예상 결과**:
- HTTP 204 No Content
- todos 테이블에서 레코드 Hard Delete
- 목록에 2개 항목만 남음

##### Step 8: 로그아웃 (10초)
**관련 기능**: F-003

1. 상단 헤더의 "로그아웃" 버튼 클릭
2. **시스템 반응**:
   - localStorage에서 토큰 삭제
   - 로그인 화면(`/login`)으로 리다이렉트

**예상 결과**:
- 로그인 화면 표시
- 인증 토큰 없음

#### 2.1.4 성공 기준
- ✅ 10분 이내에 모든 단계 완료
- ✅ 할일 3개 추가, 1개 완료, 1개 수정, 1개 삭제 성공
- ✅ 에러 없이 원활한 사용자 경험
- ✅ 모바일/데스크톱 모두 사용 가능

- 사용자가 웹 애플리케이션에 접속
- 회원가입 화면 표시

**단계**:
1. Username 입력 필드에 `testuser123` 입력
2. Password 입력 필드에 `SecurePass123!` 입력 (마스킹)
3. Email 입력 필드에 `test@example.com` 입력
4. "회원가입" 버튼 클릭
5. 로딩 스피너 표시 (0.5-1초)
6. API 응답:
   ```json
   {
     "message": "User registered successfully",
     "user": {
       "id": 1,
       "username": "testuser123",
       "email": "test@example.com"
     }
   }
   ```
7. 성공 메시지 표시: "회원가입 성공!"
8. 자동으로 로그인 화면(`/login`)으로 이동

**예상 결과**:
- HTTP 201 Created
- DB에 새 사용자 레코드 생성
- password는 bcrypt 해시로 저장 (예: `$2b$10$...`)

#### 3.1.2 대체 흐름: 중복 Username (E-001)

**단계**:
1. Username 입력: `testuser123` (이미 존재하는 사용자)
2. Password 입력: `SecurePass123!`
3. Email 입력: `test@example.com`
4. "회원가입" 버튼 클릭
5. API 응답:
   ```json
   {
     "error": "E-001",
     "message": "Username already exists"
   }
   ```
6. 화면 상단에 **빨간색 알림 배너** 표시:
   - "이미 존재하는 사용자명입니다"
   - 3초 후 자동 사라짐 또는 X 버튼으로 닫기
7. Username 필드에 빨간색 테두리 표시
8. Username 필드에 포커스 이동

**예상 결과**:
- HTTP 400 Bad Request
- 사용자는 다른 username으로 재시도

#### 3.1.3 대체 흐름: 이메일 형식 오류 (E-003)

6. 또는 서버 응답:
   ```json
   {
     "error": "E-003",
     "message": "Invalid email format"
   }
   ```

**예상 결과**:
- HTTP 400 Bad Request (서버 검증 시)
- 사용자는 올바른 이메일 형식 입력

   - 제목에 취소선: ~~세탁소 맡긴 옷 찾기~~
   - 카드 배경색 회색으로 변경

   ```http
   PATCH /api/todos/10/complete
   Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

   {
     "isCompleted": true
   }
   ```

### 3.5 할일 수정 시나리오

**관련 기능**: F-103
**관련 BR**: BR-101, BR-102, BR-103, BR-106
**화면**: 화면 4 (할일 수정 모달)

#### 3.5.1 정상 시나리오

**사전 조건**:
- 사용자가 생성한 할일 존재

**단계**:
1. 할일 항목 "분기 보고서 작성"의 "수정" 버튼 클릭
2. 할일 수정 모달 표시:
   - 제목 필드: `분기 보고서 작성` (기존 값)
   - 설명 필드: `Q4 매출 분석 및 Q1 목표 수립, 10페이지 분량` (기존 값)
   - 마감일: `2026-02-20` (기존 값)
3. 마감일 수정: `2026-02-18`로 변경 (2일 앞당김)
4. 설명 추가: `... + 경쟁사 비교 분석 포함`
5. "저장" 버튼 클릭
6. API 요청:
   ```http
   PUT /api/todos/11
   Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
   Content-Type: application/json

   {
     "title": "분기 보고서 작성",
     "description": "Q4 매출 분석 및 Q1 목표 수립, 10페이지 분량 + 경쟁사 비교 분석 포함",
     "dueDate": "2026-02-18"
   }
   ```
7. API 응답:
   ```json
   {
     "id": 11,
     "userId": 1,
     "title": "분기 보고서 작성",
     "description": "Q4 매출 분석 및 Q1 목표 수립, 10페이지 분량 + 경쟁사 비교 분석 포함",
     "dueDate": "2026-02-18",
     "updatedAt": "2026-02-11T15:30:00Z"
   }
   ```
8. 모달 닫힘
9. 목록에서 해당 항목 즉시 업데이트

**예상 결과**:
- HTTP 200 OK
- 수정된 내용 즉시 반영

#### 3.5.2 대체 흐름: 타인의 할일 수정 시도 (E-102)

**사전 조건**:
- 다른 사용자가 생성한 할일 ID를 알고 있음 (API 직접 호출 시도)

**단계**:
1. 사용자 A (userId: 1)가 사용자 B (userId: 2)의 할일 (todoId: 20) 수정 시도
2. API 요청:
   ```http
   PUT /api/todos/20
   Authorization: Bearer <User A token>
   ```
3. 서버에서 소유권 검증:
   ```javascript
   if (todo.userId !== req.user.id) {
     return res.status(403).json({
       error: "E-102",
       message: "Access denied"
     });
   }
   ```
4. API 응답:
   ```json
   {
     "error": "E-102",
     "message": "Access denied"
   }
   ```
5. 화면 알림: "접근 권한이 없습니다"

**예상 결과**:
- HTTP 403 Forbidden
- 보안 유지 (BR-102)

#### 3.5.3 대체 흐름: 존재하지 않는 할일 수정 (E-104)

**단계**:
1. 삭제된 할일 또는 잘못된 ID로 수정 시도
2. API 응답:
   ```json
   {
     "error": "E-104",
     "message": "Todo not found"
   }
   ```
3. 화면 알림: "할일을 찾을 수 없습니다"
4. 목록 새로고침 (데이터 동기화)

**예상 결과**:
- HTTP 404 Not Found

### 3.6 할일 삭제 시나리오

**관련 기능**: F-104
**관련 BR**: BR-101, BR-102
**관련 ADR**: ADR-002 (Hard Delete)
**화면**: 화면 5 (삭제 확인 다이얼로그)

#### 3.6.1 정상 시나리오 - 삭제 확인

**사전 조건**:
- 사용자가 생성한 할일 존재

**단계**:
1. 할일 항목 "세탁소 맡긴 옷 찾기"의 "삭제" 버튼 클릭 (빨간색 또는 휴지통 아이콘)
2. **확인 다이얼로그** 표시 (중앙 오버레이):
   ```
   정말 삭제하시겠습니까?

   "세탁소 맡긴 옷 찾기"

   [취소]  [삭제]
   ```
3. "삭제" 버튼 클릭 (빨간색, 경고색)
4. API 요청:
   ```http
   DELETE /api/todos/10
   Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
   ```
5. API 응답:
   - HTTP 204 No Content (응답 본문 없음)
6. 다이얼로그 닫힘
7. 해당 할일 목록에서 제거 (애니메이션: 페이드아웃 후 슬라이드업)
8. 성공 메시지 (선택): "할일이 삭제되었습니다"
9. DB에서 레코드 완전 삭제 (Hard Delete):
   ```sql
   DELETE FROM todos WHERE id = 10 AND user_id = 1;
   ```

**예상 결과**:
- HTTP 204 No Content
- 할일 영구 삭제 (복구 불가)
- 목록에서 사라짐

#### 3.6.2 대체 흐름: 삭제 취소

**단계**:
1. "삭제" 버튼 클릭
2. 확인 다이얼로그 표시
3. "취소" 버튼 클릭
4. 다이얼로그 닫힘
5. 할일 그대로 유지

**예상 결과**:
- API 호출 없음
- 실수 방지

#### 3.6.3 대체 흐름: 타인의 할일 삭제 시도 (E-102)

**단계**:
1. 사용자 A가 사용자 B의 할일 삭제 시도

## 6. 엔드투엔드 테스트 시나리오

### 6.1 시나리오 1: 신규 사용자 전체 플로우 (Cold Start)

**목표**: 처음 접하는 사용자가 회원가입부터 할일 완료까지 전체 흐름 체험

**사전 조건**: 없음

**시간**: 15분

**단계**:
1. ✅ 회원가입 (username: `new_user_001`, email: `new@test.com`)
2. ✅ 로그인
3. ✅ 빈 할일 목록 확인 ("할일이 없습니다" 메시지)
4. ✅ 할일 3개 추가:
   - "테스트 할일 1" (제목만)
   - "테스트 할일 2" (제목 + 설명)
   - "테스트 할일 3" (제목 + 설명 + 마감일)
5. ✅ 첫 번째 할일 완료 처리 (체크박스 클릭)
6. ✅ 두 번째 할일 수정 (설명 변경)
7. ✅ 세 번째 할일 삭제 (확인 다이얼로그)
8. ✅ 할일 목록에 2개 남음 (1개 완료, 1개 미완료)
9. ✅ 로그아웃
10. ✅ 재로그인 → 할일 2개 그대로 유지 확인

**성공 기준**:
- 모든 단계 에러 없이 완료
- 데이터 일관성 유지
- UI 반응 즉각적 (< 1초)

**테스트 체크리스트**:
- [ ] 회원가입 성공
- [ ] 로그인 성공
- [ ] 할일 추가 3회 성공
- [ ] 할일 완료 처리 성공
- [ ] 할일 수정 성공
- [ ] 할일 삭제 성공 (확인 다이얼로그 표시)
- [ ] 로그아웃 성공
- [ ] 재로그인 후 데이터 유지 확인

---

### 6.2 시나리오 2: 기존 사용자 일상적 사용 (Daily Routine)

**목표**: 이미 계정이 있는 사용자의 일상적인 할일 관리 시뮬레이션

**사전 조건**:
- 기존 사용자 계정 존재 (username: `existing_user`)
- 이미 할일 5개 등록되어 있음

**시간**: 10분

**단계**:
1. ✅ 로그인
2. ✅ 기존 할일 5개 확인
3. ✅ 오늘 완료한 할일 2개 체크박스 클릭
4. ✅ 새로운 할일 추가: "긴급 회의 준비"
5. ✅ 마감일 지난 할일 1개 마감일 연장 (수정)
6. ✅ 더 이상 필요 없는 할일 1개 삭제
7. ✅ 완료된 할일 목록 확인 (취소선 표시)
8. ✅ 로그아웃

**성공 기준**:
- 빠른 완료 체크 (< 0.5초 응답)
- 수정/삭제 즉시 반영
- 사용자는 스트레스 없이 일상적으로 사용 가능

**테스트 체크리스트**:
- [ ] 로그인 성공
- [ ] 기존 할일 목록 표시
- [ ] 완료 처리 2회 성공
- [ ] 새 할일 추가 성공
- [ ] 할일 수정 (마감일 연장) 성공
- [ ] 할일 삭제 성공
- [ ] 완료/미완료 시각적 구분 확인
- [ ] 로그아웃 성공

---

### 6.3 시나리오 3: 에러 복구 시나리오 (Error Handling)

**목표**: 모든 예외 상황에서 적절한 에러 처리 및 복구 확인

**사전 조건**: 테스트용 계정 (username: `test_error_user`)

**시간**: 10분

**단계**:

**Phase 1: 회원가입 에러**
1. ✅ 중복 username으로 회원가입 시도 → E-001 에러 표시
2. ✅ 잘못된 이메일 형식으로 시도 → E-003 에러 표시
3. ✅ 올바른 정보로 재시도 → 성공

**Phase 2: 로그인 에러**
4. ✅ 잘못된 비밀번호로 로그인 시도 → E-002 에러 표시
5. ✅ 올바른 비밀번호로 재시도 → 성공

**Phase 3: 할일 관리 에러**
6. ✅ 제목 없이 할일 추가 시도 → E-103 에러 표시
7. ✅ 제목 입력 후 재시도 → 성공
8. ✅ 브라우저 DevTools로 토큰 삭제 후 할일 추가 시도 → E-101 에러, 로그인 화면으로 리다이렉트
9. ✅ 재로그인 후 정상 사용

**Phase 4: 권한 에러 (고급)**
10. ✅ Postman/Thunder Client로 타인의 할일 ID에 대해 수정 API 직접 호출 → E-102 에러
11. ✅ 존재하지 않는 할일 ID로 삭제 시도 → E-104 에러

**성공 기준**:
- 모든 에러 코드 (E-001~E-104) 정확히 표시
- 에러 후 복구 가능 (입력 데이터 보존)
- 보안 에러 (E-101, E-102) 적절히 차단

**테스트 체크리스트**:
- [ ] E-001: 중복 username 에러 확인
- [ ] E-002: 잘못된 자격증명 에러 확인
- [ ] E-003: 이메일 형식 에러 확인
- [ ] E-101: 미인증 접근 에러 확인 (로그인 리다이렉트)
- [ ] E-102: 권한 없는 수정/삭제 에러 확인
- [ ] E-103: 제목 필수 에러 확인
- [ ] E-104: 존재하지 않는 할일 에러 확인
- [ ] 모든 에러 후 복구 가능

---

### 6.4 시나리오 4: 반응형 디자인 테스트 (Cross-Device)

**목표**: 다양한 화면 크기에서 UI 정상 표시 확인

**사전 조건**: 테스트용 계정

**시간**: 5분

**단계**:

**Desktop (1920x1080)**:
1. ✅ Chrome DevTools → 1920x1080 해상도 설정
2. ✅ 로그인 화면 표시 확인 (중앙 정렬 카드)
3. ✅ 할일 목록 그리드 레이아웃 확인 (2-3열)
4. ✅ 모달 크기 적절 (화면의 50% 정도)

**Tablet (768x1024)**:
5. ✅ DevTools → iPad 해상도 설정
6. ✅ 할일 목록 2열 또는 1열 레이아웃
7. ✅ 터치 가능한 버튼 크기 확인

**Mobile (375x667)**:
8. ✅ DevTools → iPhone SE 해상도 설정
9. ✅ 할일 목록 세로 스택 (1열)
10. ✅ 모달 전체 화면 또는 하단 시트 형태
11. ✅ 키보드 올라와도 입력 필드 가려지지 않음
12. ✅ 스크롤 원활

**성공 기준**:
- 모든 해상도에서 UI 깨지지 않음
- 텍스트 가독성 유지
- 버튼 클릭/터치 가능

**테스트 체크리스트**:
- [ ] Desktop (1920x1080) 정상 표시
- [ ] Tablet (768x1024) 정상 표시
- [ ] Mobile (375x667) 정상 표시
- [ ] 가로 모드도 테스트 (선택)
- [ ] 모든 해상도에서 기능 동작

## 8. 부록

### 8.1 사용자 시나리오 작성 시 고려사항

**실제 사용자 관점**:
- 기술 용어 최소화 (사용자는 "API", "JWT"를 모름)
- 사용자가 보는 화면 중심 서술
- 구체적인 행동 동사 사용 (클릭, 입력, 확인 등)

**PRD와의 일관성**:
- 모든 시나리오는 PRD의 기능 ID (F-001 등) 참조
- 에러 코드 (E-001 등) 정확히 매칭
- UI 화면 번호 (화면 1-5) 명시

**테스트 가능성**:
- 사전 조건 명확히 정의
- 단계별로 검증 가능한 예상 결과
- HTTP 상태 코드 및 응답 예시 포함

### 8.2 에러 코드 빠른 참조

| 코드 | 메시지 (영문) | 메시지 (한글) | HTTP |
|------|---------------|---------------|------|
| E-001 | Username already exists | 이미 존재하는 사용자명입니다 | 400 |
| E-002 | Invalid credentials | 아이디 또는 비밀번호가 일치하지 않습니다 | 401 |
| E-003 | Invalid email format | 올바른 이메일 형식이 아닙니다 | 400 |
| E-101 | Authentication required | 로그인이 필요합니다 | 401 |
| E-102 | Access denied | 접근 권한이 없습니다 | 403 |
| E-103 | Title is required | 할일 제목은 필수 항목입니다 | 400 |
| E-104 | Todo not found | 할일을 찾을 수 없습니다 | 404 |
